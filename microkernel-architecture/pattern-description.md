# توضیح الگو



```
رایج ترین الگوی معماری، الگوی معماری لایه ای است 
 که همچنین الگوی معماری n-tier شناخته می‌شود.
  این الگو عملا بهترین استاندارد برای اکثر برنامه های Java EE است
   و ازاین‌رو به طور گسترده توسط اکثر معماران، طراحان و توسعه‌دهندگان شناخته شده‌است.
الگوی معماری لایه‌ای با ارتباطات سنتی فناوری اطلاعات و ساختارهای سازمانی موجود
 در آن مطابقت دارد.
 اکثر شرکت ها، آن را به یک انتخاب برای اکثر برنامه‌های تجاری خود تبدیل
  می کنند.



```

```
اجزای درون الگوی معماری لایه ای به لایه های افقی، سازماندهی شده اند
که هر لایه نقش خاصی را در درون برنامه انجام می دهد
(به عنوان مثال، منطق ارائه یا منطق تجاری).
اگر چه الگوی معماری لایه ای تعداد و انواع لایه هایی
که باید در الگو وجود داشته باشد را مشخص نمی کند،
بیشتر معماری های لایه ای از چهار لایه استاندارد تشکیل شده اند:
 ارائه، کسب و کار، پایداری و پایگاه داده.
  در بعضی از موارد، لایه کسب و کار و لایه پایداری
  در یک لایه تجاری واحد ترکیب می شوند.
به ویژه هنگامی‌که منطق پایداری(به عنوان مثال، SQL یا HSQL)
 در اجزای لایه تجاری تعبیه شده‌است.
بنابراین، برنامه ها کوچکتر ممکن است فقط سه لایه داشته باشند،
 در حالی که برنامه های بزرگتر و
برنامه های تجاری پیچیده تر ممکن است شامل پنج لایه یا بیشتر باشد.
```

```
هر لایه از الگوی معماری لایه ای نقش خاصی و
مسئولیتی در درون برنامه دارد.
به عنوان مثال، لایه ارائه مسئول رسیدگی به تمام رابط کاربری و
منطق ارتباط مرورگر است،
در حالی که لایه تجاری مسئول اجرای قوانین تجاری خاص مرتبط با
درخواست است.
 هر لایه در معماری یک انتزاع را در اطراف تشکیل می دهد
کاری که باید برای برآورده کردن یک درخواست تجاری خاص انجام شود.
 برای مثال، لایه ارائه نیازی به دانستن ندارد
یا نگران نحوه دریافت اطلاعات مشتری باشید. فقط باید نمایش داده شود
آن اطلاعات روی صفحه نمایش با فرمت خاص. به طور مشابه،
لایه تجاری نیازی به نگرانی در مورد نحوه قالب بندی ندارد
داده های مشتری برای نمایش روی صفحه یا حتی جایی که مشتری است
داده ها از فقط باید داده ها را از persis بدست آورد
لایه tence، منطق تجاری را در برابر داده ها انجام دهید (مثلاً محاسبه کنید
مقادیر یا داده های انبوه)، و آن اطلاعات را به پیش
لایه ارسال
```

```
یکی از ویژگی های قدرتمند الگوی معماری لایه ای است
تفکیک نگرانی ها بین اجزاء اجزای داخل
یک لایه خاص فقط با منطق مربوط به آن لایه سروکار دارد. برای
به عنوان مثال، مؤلفه‌ها در لایه ارائه فقط با قبل سروکار دارند.
منطق ارسال، در حالی که اجزای موجود در لایه تجاری
فقط با منطق تجاری برخورد کنید این نوع طبقه بندی اجزا
ساختن نقش های موثر و مدل های مسئولیت پذیری را آسان می کند
معماری شما، و همچنین توسعه، آزمایش، حکومت،
و حفظ برنامه های کاربردی با استفاده از این الگوی معماری به دلیل
رابط های مؤلفه به خوبی تعریف شده و دامنه اجزای محدود.
```
